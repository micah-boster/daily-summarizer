<action>
**Pipeline ordering fix (src/pipeline_async.py):**

At approximately lines 552-568, the pipeline currently calls:
```python
attribution_result = _attribute_entities(synthesis_result, current, ctx.config)
# ... (intervening code)
_discover_and_register_entities(synthesis_result, current, ctx.config)
```

Swap the order so `_discover_and_register_entities` runs FIRST, then `_attribute_entities` runs AFTER. This ensures newly-seen entities are in the registry when attribution runs, so first-mention entities get attributed on the same pipeline run.

Both calls are already wrapped in independent try/except blocks, so the swap is safe — neither depends on the return value of the other for control flow. The only semantic dependency is that discovery populates the registry before attribution reads it.

Find the exact call sites by grepping for `_attribute_entities` and `_discover_and_register_entities` in the pipeline execution section (around line 550). Move the entire `_discover_and_register_entities(...)` call block (including its try/except wrapper) to BEFORE the `_attribute_entities(...)` call block.

**Template path fix (src/entity/views.py):**

In the `generate_entity_report` function (around line 263), the default parameter is:
```python
template_dir: str = "templates",
```

Change the default to resolve against the project root using Path:
```python
template_dir: str = str(Path(__file__).parents[2] / "templates"),
```

Add `from pathlib import Path` at the top if not already imported. The `parents[2]` navigates from `src/entity/views.py` up to the project root (views.py -> entity/ -> src/ -> project root).

Verify `Path(__file__).parents[2]` resolves correctly by checking the file's location in the directory tree: `src/entity/views.py` — parents[0]=`src/entity/`, parents[1]=`src/`, parents[2]=project root.
</action>
