In src/entity/merger.py, implement:
- score_pair(name_a: str, name_b: str) -> float: normalize_for_matching both names, return fuzz.token_sort_ratio
- generate_proposals(repo: EntityRepository, entity_type: str | None = None, limit: int = 10) -> list[dict]:
  1. List active entities (WHERE deleted_at IS NULL AND merge_target_id IS NULL), filtered by entity_type if provided
  2. Group by entity_type (process each type separately)
  3. For each pair in group, check if proposal already exists (any status) via repo.get_existing_proposals -- skip if exists
  4. Score pair via score_pair -- skip if < 80
  5. Build proposal dict: {source_entity, target_entity, score, source_mention_count, target_mention_count, source_mention_types, target_mention_types}
  6. Sort by score descending
  7. Return first `limit` proposals

Use imports: from rapidfuzz import fuzz, from src.entity.normalizer import normalize_for_matching

Each proposal dict should contain:
  - source_entity: Entity object (less-mentioned entity)
  - target_entity: Entity object (more-mentioned entity, will be canonical if accepted)
  - score: float (similarity score)
  - source_context: {mention_count: int, source_types: list[str]}
  - target_context: {mention_count: int, source_types: list[str]}

When building proposals, assign source/target so that the entity with MORE mentions is always `target_entity` (the surviving canonical). This implements "most-mentioned name becomes canonical automatically" from CONTEXT.md.

   ```python
   review_p.add_argument("--limit", type=int, default=10, help="Max proposals per session (default 10)")
   review_p.add_argument("--type", dest="entity_type", default=None,
                          choices=["partner", "person"], help="Filter by entity type")
   ```

2. In handle_entity_command(), add case for cmd == "review": call _cmd_review(repo, args)

3. Implement _cmd_review(repo, args):
   a. Import generate_proposals from src.entity.merger
   b. Call generate_proposals(repo, entity_type=args.entity_type, limit=args.limit)
   c. If no proposals: print "No merge proposals found." and return
   d. Print header: "Found {N} merge proposal(s). Reviewing..."
   e. Track counters: accepted=0, rejected=0, skipped=0
   f. For each proposal, print compact display:
      ```
      --- Proposal {i}/{total} (score: {score:.0f}) ---
      A: {source_entity.name} ({source_entity.entity_type})
         Mentions: {source_context.mention_count} from {', '.join(source_context.source_types) or 'none'}
      B: {target_entity.name} ({target_entity.entity_type})
         Mentions: {target_context.mention_count} from {', '.join(target_context.source_types) or 'none'}
      Canonical if merged: {target_entity.name}
      ```
   g. Prompt: input("[a]ccept / [r]eject / [s]kip / [q]uit: ").strip().lower()
   h. Handle responses:
      - "a" or "accept": Save proposal as approved via repo.save_proposal(source_id, target_id, reason=f"similarity score {score:.0f}", status="approved"). Increment accepted.
      - "r" or "reject": Save proposal as rejected via repo.save_proposal(source_id, target_id, reason=f"rejected by user", status="rejected"). Increment rejected.
      - "s" or "skip": Increment skipped. Continue to next. (No row saved -- can be proposed again.)
      - "q" or "quit": Break out of loop.
      - Any other input: Print "Invalid choice. Use a/r/s/q." and re-prompt same proposal.
   i. After loop, print summary:
      ```
      Review complete: {accepted} accepted, {rejected} rejected, {skipped} skipped
      ```
      If accepted > 0: print "Run merge execution to apply accepted proposals." (Note: merge execution is Plan 22-02)

4. Update the else clause in handle_entity_command to include "review" in usage string.

Add test for CLI registration:
- test_review_subparser_registered: Verify argparse accepts "entity review --limit 5 --type partner" without error.

Note: Full interactive testing of input() is deferred -- the core logic is covered by merger.py tests. CLI integration is validated by parsing and function dispatch.
